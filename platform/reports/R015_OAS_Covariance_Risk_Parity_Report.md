# R015：基于 Oracle Approximating Shrinkage (OAS) 协方差估计的风险平价策略实验报告

## 1. 实验背景与假设

### 1.1 实验背景
在多资产风险均衡配置（Risk Parity）中，历史协方差矩阵（Covariance Matrix）是分配权重的核心输入。然而，在滚动窗口天数（T=120）较少或资产维度（N）较多时，常规的样本协方差矩阵往往包含巨大的随机噪声和偏误。这导致逆协方差矩阵的求解对历史收益的极小变化高度敏感，从而引起资产权重剧烈抖动，产生不必要的交易摩擦成本。

在此前的 R001 研究中，曾引入 Ledoit-Wolf 协方差收缩法来平滑历史噪声，获得了较好的业绩。然而，在小样本高斯分布的假设下，Ledoit-Wolf 收缩所采用的收缩系数并非均方误差（MSE）最小化意义下的理论最优解。

### 1.2 核心假设
本实验（R015）提出 **Oracle Approximating Shrinkage (OAS) 协方差估计风险平价策略**。
OAS 是一种专门针对高斯样本设计的协方差估计收缩方法，它在均方误差（MSE）最小化的目标下，通过最优收缩系数的闭式解公式，将样本协方差 $S$ 向常数方差、零相关性的目标矩阵 $F = \mu I$ 收缩。
我们假设，使用 OAS 替代 Ledoit-Wolf 收缩协方差，能在中短滚动周期下提供比 Ledoit-Wolf 更具鲁棒性的风险平价求解权重，从而在全基线多资产配置中实现夏普比率的进一步提升以及回撤的平滑。

## 2. 策略代码实现与注册
我们已在 `platform/src/platform_core/strategy.py` 中实现了 `RiskParityOASCovStrategy`（注册键为 `risk_parity_oas_cov`）：
- **OAS 协方差估计**：使用 NumPy 计算样本去均值收益率的方差、迹 $	ext{tr}(S)$ 以及平方和 $	ext{tr}(S^2)$，闭式解估计最优收缩系数 $ho$ 并完成线性收缩。
- **权重求解**：使用收敛速度快且 100% 稳定的循环坐标下降（CCD）算法进行风险平价求解。
- **策略注册**：将新策略类注册至内置的 `BUILTIN_STRATEGIES` 中。
*(注：后续由于回测未通过准入审计，该段策略代码已被物理擦除)*

## 5. 绩效归因与过拟合审计

### 5.1 整体绩效改善归因（局部优势与大面积退化）
从 12 个典型基准配置的测试结果来看，OAS 协方差估计策略并未表现出全线优化的特性：
1. **仅在等权重和极窄资产配置下有效**：策略在等权基准（`baseline_mvp_equal_weight`）和基线 `baseline_m3m4_fundamental` 下夏普显著改善，主要是因为窄资产（仅3只资产）在无负相关性干扰的情况下，OAS 能比样本方差更佳地平滑小样本极值。
2. **大面积的业绩退化**：在 9 个典型滚动和 EWMA 估计的境内外多资产组合中，**夏普比率发生了全线下降**。最显著的为全球红利组合（夏普由 1.531 下降至 1.320）。同时，多资产组合的最大回撤全线扩大，说明在多资产配置中，OAS 协方差估计的效果甚至不如基线。
3. **退化原因分析**：
   OAS 收缩的目标矩阵是 $F = \mu I$（对角等方差矩阵），其将资产间所有的互协方差（相关系数）都无区别地朝着 0 进行线性收缩。这种设计没有考虑到资产组合中原本非常有用的低相关或强负相关结构（如股债的弱相关性、黄金与股票的负相关性）。在多资产配置组合中，这种粗暴地削弱全部相关性的收缩模式（破坏了分散化效应的分母）导致等风险贡献分配权重向错误的防御端转移，从而造成了严重的牛市踏空与配置失效。

### 5.2 换手率摩擦审计
审计显示，在 HRP 多基准下，年化换手率出现了 **+35.80%** 的上升，超出了 30% 的换手审计阈值。这说明在资产相关系数被错误朝着 0 收缩的过程中，局部估计微调的敏感性导致了组合在调仓日过度换仓，产生了显著的样本外摩擦风险。

### 5.3 参数样本充足率审计
策略参数的估计虽然不涉及复杂的极大似然估计，但因对多资产相关系数的物理降噪模式存在系统性设计缺陷，造成样本外泛化风险，未能通过鲁棒性检验。

## 6. 实验结论与推荐动作

* **研究成果判定**：差异不大 / 有局部优势 / 表现平庸 (物理拒绝)
* **推荐动作**：
  1. **物理拒绝合入平台主干**：由于该策略无法在多资产大类配置下实现全线夏普优化，且最大回撤普遍变大，部分配置甚至违反了换手率审计红线，不符合平台策略的稳健防御性原则。
  2. **代码完全清除**：目前已从 `platform/src/platform_core/strategy.py` 中物理擦除全部该策略的相关实现代码和 `BUILTIN_STRATEGIES` 中的注册。保持代码库和默认配置文件的绝对干净。
